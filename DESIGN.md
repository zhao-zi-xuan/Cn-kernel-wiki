# DESIGN — 设计说明（关键决策与理由）

> 这份文档解释 NPUKernelWiki 的**设计决策为什么这么定**，而不是它做了什么（那是 [README](README.md) /
> [WORKLOG](WORKLOG.md) 的事）。核心论点：**这是一次方法论驱动的知识工程实践，不是把 PR 摘要堆在一起。**
> 每条决策都给出"决策 → 为什么 → 取舍/代价"，且每条都由 `scripts/validate.py` 或流程强制，不是口头约定。

---

## 0. 一句话定位

把 MIT HAN Lab 的 KernelWiki（NVIDIA kernel 知识库）的**方法论**——三层架构、schema 驱动、
LLM 可检索——迁移到**国产 NPU（昇腾优先）**。价值不在"又抄了一个 wiki"，而在于：
面对一个资料零散、无统一硬件、闭源成分多的国产生态，**用一套可验证的工程约束把知识组织成
可溯源、可检索、不失真的结构**。下面四组决策就是这套约束的骨架。

---

## 1. 三层架构 + schema + validator：为什么要"自找麻烦"

**决策**：`sources/`（不可变原始摘要）→ `wiki/`（综合知识页）→ `queries/`（自动索引），
每页带 YAML frontmatter、靠唯一 `id` 互引，改完必须 `validate.py` 跑出 **0 errors** 才算完成。

**为什么**：知识库最大的失败模式是**随规模腐化**——链接失效、口径不一、来源丢失、越写越像一堆
笔记。把"原始事实"和"综合结论"物理分层，再用 schema + 引用完整性校验当**强制函数（forcing
function）**，就能在每一次提交时机械地挡住腐化。这是"知识工程"区别于"写文档"的地方：
**正确性是被校验出来的，不是靠自觉。**

**取舍**：每加一页都要过 schema、对齐词表、保证引用的 id 存在——写起来更慢。但换来的是
任意时刻整库自洽，且能被工具（`query.py`/`get_page.py`）和 LLM 稳定检索。

---

## 2. 铁律 D1 —— Ascend-first（昇腾优先）

**决策**：昇腾为主；寒武纪/摩尔/海光等只做 interface-level 提及。任何 `architectures` 含
secondary vendor 却不含 `ascend-*`/`davinci` 的 wiki 页，**必须带 `secondary_vendor_note`**，
否则 validator 报错。

**为什么**：
- **聚焦即深度**。国产 NPU 生态分散，若"雨露均沾"，每家都只能浅尝，最后是一份谁都不解决问题的
  概览。选昇腾是因为它**生态最完整、公开材料最多**（AscendC 官方文档、vllm-ascend 活跃上游），
  能支撑到"具体 kernel 怎么写"的深度。
- **诚实边界**。别家资料我拿不到一手证据，与其编，不如用 `secondary_vendor_note` **显式标注
  "这里只是接口层提及，未经昇腾级别的核实"**，把知识的可信度边界写进 schema。

**取舍**：牺牲横向覆盖，换纵向深度和可信度。这是**主动的范围选择**，不是能力缺失——而且它被
校验强制，不会因为后面想"顺手加一句别家"就悄悄破坏定位。

---

## 3. 铁律 D2 —— No-hardware（无真实硬件）

**决策**：复现性（`reproducibility`）**封顶 `snippet`**（禁止 `runnable`/`benchmarked`）；
kernel 页每条 `performance_claims` 的 `measurement` **只能 `source-reported` 或 `official-doc`**
（`self-measured` 被 schema 禁止）。validator 逐条检查。

**为什么**：
- **没有昇腾卡，就没有资格给性能背书**。学术和工程诚信的底线是不拿没有的数据充数。与其含糊，
  不如把这条**写死进数据模型**：所有性能都必须挂一个 `source_id` 指向真实来源，且标明是"别人报告的"
  而非"我测的"。
- 实际执行中这条被反复触发：本项目 10 个 kernel 页引用的源 PR **几乎都没披露绝对数字**，于是所有
  `performance_claims` 都写成**定性的 source-reported**（如"合并 token+scale 传输以减少 bubble，
  未披露绝对数值"）。**一个性能数字都没有编造**——这正是 D2 想保证的。

**取舍**：页面读起来不如"提速 3.2×"那么有冲击力。但**可信 > 好看**：这份知识库敢让人顺着
`source_id` 去核对每一句性能话术。

---

## 4. 范围边界 —— 为什么不做分布式（HCCL / 超节点 / 大 EP 通信）

**决策**：明确**排除**分布式系统话题；候选账本里 HCCL/超节点/跨卡通信类 PR 一律 exclude。

**为什么**：
- **单卡 kernel 与集群通信是两门学问**。kernel 优化关心的是 Cube/Vector/UB/MTE 这条**片内数据通路**；
  分布式关心的是 HCCL 拓扑、超节点、EP 通信调度——两者的知识结构、证据来源、评价标准都不同。混在一起
  只会让"kernel 优化知识库"这个定位失焦。
- **可判定的纳入规则**。范围边界不是模糊偏好，而是账本里一条**可操作的判据**：只有改动
  `.cce`/`csrc/kernels/`/`op_kernel/` 这类 on-device kernel 的 PR 才 include；host 侧调度、
  模型接线、通信一律不进。这让"该不该收"有据可依，而不是每次拍脑袋。

**取舍**：放弃了一大块热门话题（大 EP、超节点是当前工程热点）。但**清晰的 scope 是知识库可用性的
前提**——用户知道来这里能得到什么、不会得到什么。

---

## 5. 两段式摄入 —— 为什么不直接抓 PR 建页

**决策**：摄入分两阶段。**Stage-1 候选账本**（`candidates/*.yaml`）：把搜到的 PR 逐条标
`include` / `defer` / `exclude` + reason；**Stage-2 生成**（`generate-pr-pages.py`）：只有 include
行才拉成 `sources/` 页。

**为什么**：
- **决策与执行分离，且决策留痕**。"这个 PR 该不该进、为什么"是**编辑判断**，值得单独固化下来；
  账本让每条判断都带 reason、可复查、可翻案。本项目实际把 86 个 PR 过成 **31 include / 37 defer /
  18 exclude**，每条都有理由——这本身就是一份**可审计的知识策展记录**。
- **抗噪声、可迭代**。第一遍分类靠标题会误判（例如几个 `model_runner_v2` PR 标题写着"优化某 kernel"，
  但完整 diff 只改了 host 调用侧）。两段式让我可以**先粗分、再用完整文件列表收紧**，把这类灰色地带
  降级为 defer 而不污染 sources 层。一步到位的抓取做不到这种"留有余地的修正"。

**取舍**：多一层账本要维护。但换来的是 sources 层的**信噪比**——进入知识库的每一页都经过明确判定，
而不是"搜到什么算什么"。

---

## 6. 这体现的知识工程价值（给评审的总结）

| 别人可能以为 | 实际做的 |
|---|---|
| 抓一堆 PR 摘要拼成 wiki | 用 schema + validator 把正确性**机械强制**，0-errors 是完成判据 |
| 昇腾资料多所以只写昇腾 | Ascend-first 是**主动的深度/诚信取舍**，且边界写进校验 |
| 性能数字随便写写 | D2 禁止自测/编造，所有性能挂 `source_id`，**实际一个数字没编** |
| 什么 kernel 都收 | 可判定的纳入规则 + 两段式账本，**31/37/18 的可审计策展** |
| 页面各自为战 | 五类页（hardware/technique/kernel/pattern/language）靠 `id` 互引成图，
  再加 `references/primer.md` 按意图导航 |

一句话：**约束优先于产出**。先把"什么算对、什么算可信、什么算在范围内"用可执行的规则定死，
再在规则内填内容——这才是把领域知识做成**工程制品**而非笔记合集的关键。
